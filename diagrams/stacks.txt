I ran a few traces. Below are my findings.
Nasko, could you review and comment on proposed solution?
Repro code:
===
<script> function go(){
 var x=window.open('https://www.google.com/csi');
 setTimeout(function(){
    x.alert('Alert');
    x.prompt('E-mail','');
    x.prompt('Password','');} ,1000);
}
</script>
===
The script first opens new window and retains reference to it. After a timeout the script uses the new window refernce to call alert and then prompts.

While alert is displayed the user can interact with URL bar and start navigation. This results in the following calls:
- RenderFrameHostImpl::ProcessBeforeUnloadCompleted
- NavigationRequest::BeginNavigation
- WebContentsImpl::DidStartNavigation
This call closes currently opened alert, but the javascript code opens a prompt and somehow blocks navigation. This creates illusion that the prompt comes from the site the user navigates to (URL bar shows the URL, user just typed in).

If navigation continued, like it does on desktop, then RenderFrameHostImpl::DidCommitNavigation would get called. It in turn would call RenderFrameHostManager::UnloadOldFrame which closes all dialogs. There is also some logic that I don't understand that prevents the rest of javascript form executing.

I looked at behavior in Safari. It dismisses the alert and never shows the prompt when user initiates navigation. I think this is correct behavior. One way to achieve this could be to check if request to display dialog comes from the same origin that is currently displayed. This way initiating navigation would change the origin and it wouldn't match the one reproted by javascript. This is just a speculation as I'm unfamiliar with the code.
